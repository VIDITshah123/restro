# Database Architecture & Storage Mapping
**Byte Cafe: AI-Based Smart Restaurant Management System**

This document provides a highly detailed breakdown of all files and folders related to data storage, schema design, and database connectivity across the entire application ecosystem.

---

## 1. Database Configuration & Schema (`server/db/`)
This folder acts as the brain for the Node.js backend to connect, configure, and structure the database.

* **`server/db/index.js`**  
  **Purpose:** The Connection Initializer.  
  **Details:** This file uses the `better-sqlite3` driver to establish a synchronous connection to the database. It is highly critical because it executes the `PRAGMA journal_mode = WAL;` command upon booting. By enabling **Write-Ahead Logging (WAL)**, it ensures the database doesn't lock when multiple customers try to place an order at the exact same millisecond.

* **`server/db/migrations/` (Folder)**  
  **Purpose:** Version Control for Database Structure.  
  **Details:** Contains raw SQL files that map out the architecture of the application.
  * **`001_init.sql`**: The master schema file. It contains the exact `CREATE TABLE` logic and enforces Foreign Key constraints for:
    * `users` (Admin/Waiter accounts)
    * `tables` (Physical restaurant seating)
    * `categories` & `menu_items` (Food catalog)
    * `variants` (Customizations like 'Extra Cheese')
    * `orders` & `order_items` (Transactional data & Profit snapshots)

---

## 2. Physical Storage & Memory (`server/data/`)
This folder contains the actual physical files generated by SQLite to store the live production data on the server's hard drive.

* **`server/data/restaurant.db`**  
  **Purpose:** The Master Storage File.  
  **Details:** This is the primary physical file where all permanent data (menus, orders, financial analytics) is stored. Because we use SQLite, we do not need a massive external database server (like MongoDB or PostgreSQL). The entire database lives compactly inside this single, highly portable file.

* **`server/data/restaurant.db-wal`**  
  **Purpose:** The Write-Ahead Log (High Concurrency Engine).  
  **Details:** When a customer places an order, the system does not write it directly to `restaurant.db` (which would cause a lock). Instead, it rapidly writes the transaction to this `-wal` file. This allows background processes to safely merge the data later while the system continues serving other customers instantly.

* **`server/data/restaurant.db-shm`**  
  **Purpose:** The Shared Memory Index.  
  **Details:** A temporary file automatically created by SQLite when WAL mode is enabled. It acts as a lightning-fast index that helps the server quickly locate specific transactions inside the `-wal` file.

---

## 3. The AI Microservice Connection (`ai-service/`)
Because the Python AI engine runs as a completely separate microservice, it requires its own secure way to access the database without causing conflicts with the Node.js API.

* **`ai-service/db.py`**  
  **Purpose:** The Python SQLite Bridge.  
  **Details:** This script uses Python's built-in `sqlite3` library to establish a connection to the exact same `server/data/restaurant.db` file.  
  **Security & Performance:** To ensure the AI never accidentally deletes an order or corrupts financial data, this connection is typically configured for **Read-Only** access. It simply reads the historical `order_items` data to build its mathematical frequency matrices for the "Top Picks" collaborative filtering engine.
